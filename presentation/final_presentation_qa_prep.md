### Q: What constraints (data, approvals, environments) could block adoption — and how are they addressed?

**ANSWER (Velen / YJ):**

> "The main constraint is **Graph API admin consent** — a tenant admin must grant the app registration five read-only permissions: SecurityEvents.Read.All, SecurityActions.Read.All, InformationProtection.Read.All, Policy.Read.All, and Reports.Read.All.

## Category 2: Integration with Azure & Microsoft Solutions (25 pts)

### Q: Which Microsoft services are integrated — and why they are essential?

**ANSWER (Velen / YJ — architecture walkthrough):**

> "We integrate **9 Azure and Microsoft services**, and every one of them is load-bearing — not decorative:
>
> 1. **Microsoft Graph Security API** — the data source. All 4 assessment tools (Secure Score, Defender, Purview, Entra) query Graph. Without it, there's no assessment.
> 2. **Azure OpenAI (GPT-4o)** — the reasoning engine. Generates remediation plans with prioritization, effort estimates, and PowerShell scripts. We support **multi-model selection** (GPT-4o and GPT-4o-mini).
> 3. **Azure AI Content Safety** — RAI gate. Every LLM input and output is screened. Blocks prompt injection attempts and unsafe content.
> 4. **Entra ID** — identity layer. OAuth2 authorization code flow for user auth, delegated permissions for Graph, Managed Identity for service-to-service.
> 5. **Azure Key Vault** — secrets management. Graph API credentials, client secrets — accessed via Managed Identity, never hardcoded.
> 6. **Azure Container Apps** — deployment target. Scale-to-zero, health probes, Managed Identity built-in.
> 7. **Azure Container Registry** — image store for CI/CD pipeline.
> 8. **Azure Application Insights** — distributed tracing. Every tool call is an OpenTelemetry span with duration, token usage, and Graph API latency.
> 9. **Fabric Lakehouse** — longitudinal telemetry. After each assessment, posture snapshots are pushed for trend analysis and Power BI dashboards."

### Q: What is the minimum integration path to still deliver value?

**ANSWER (Velen / YJ):**

> "The absolute minimum is: **Graph Security API + Azure OpenAI + Entra ID**. That gives you the assessment tools, the reasoning for remediation plans, and the auth to access tenant data.
>
> In fact, we designed the agent to work in a **progressive enhancement** model. If Azure AI Content Safety is unavailable, the agent still functions — it just skips the RAI filter (logged as a warning). If the Fabric endpoint isn't configured, `push_posture_snapshot` returns a no-op success. If Key Vault is unreachable, the readiness probe reports it but the agent can still use environment-variable credentials for development.
>
> For a zero-cloud local development experience: `pip install`, set zero environment variables, and the agent runs with mock data — every tool returns realistic simulated responses. This makes it easy for any developer to start contributing without an Azure subscription."

---

### Q: How does the solution degrade gracefully when integrations fail?

**ANSWER (Velen / YJ):**

> "We designed explicit fallback paths for every integration:
>
> - **Graph API returns 403** (missing permission): The tool returns partial data with a `permissions_note` field explaining what's missing and how to fix it. The agent includes this in its response — it doesn't crash or hallucinate.
> - **Azure OpenAI is unavailable**: The remediation plan tool returns a structured error. The web agent surfaces a helpful message instead of a stack trace.
> - **Content Safety endpoint is down**: The middleware logs a warning and allows the request through. We chose availability over blocking — but the audit log records that the safety check was skipped.
> - **Fabric Lakehouse not configured**: The `push_posture_snapshot` tool returns `{"status": "skipped", "reason": "No Fabric endpoint configured"}`. No crash.
> - **Readiness probe (`/ready`) reports all dependency status**: If Graph auth is expired, OpenAI is unreachable, or Key Vault can't be accessed, the probe returns 503 with a detailed status JSON telling you exactly which dependency is down."

### Q: Can integrations be swapped/extended without rewriting core logic?

**ANSWER (Velen / YJ):**

> "Yes — this was a deliberate architectural decision. The tool system is **completely modular**:
>
> - Each tool is a standalone Python module in `src/tools/` with a standard async function signature.
> - Tool schemas are defined in `src/tools/definitions.py` — shared between the CLI agent and the web agent.
> - Adding a new tool requires changes in exactly **4 files**: implement the tool, add the schema, wire the dispatcher, and update the system prompt. We documented this in our `docs/web-agent-tool-guide.md`.
>
> The LLM backend is swappable — we've already implemented multi-model support. The `AVAILABLE_MODELS` config lets operators configure which Azure OpenAI deployments are available. The Copilot SDK runtime also supports BYOM (Bring Your Own Model) via a provider configuration — you could point it at a different model endpoint without changing any tool code.
>
> The identity layer is also abstracted — `azure-identity` supports `DefaultAzureCredential` which auto-detects whether you're running locally (interactive browser), in CI (federated OIDC), or in production (Managed Identity)."

## Category 4: Security, Governance & Responsible AI (15 pts)

### Q: Data handling: least privilege, minimization, retention, isolation?

**ANSWER (Velen / YJ):**

> "Every design choice was least-privilege-first:
>
> - **Graph API scopes**: Five read-only permissions. No write permissions. No admin-level permissions. The agent uses delegated permissions — it can only see what the authenticated user can see.
> - **Data minimization**: Before any data reaches Azure OpenAI, our PII redaction middleware removes tenant GUIDs, user emails, UPNs, IP addresses, and display names. The LLM sees `[TENANT_ID]`, `[USER_EMAIL]`, etc. After the model responds, we re-hydrate for display if needed.
> - **Retention**: Audit logs have a 90-day retention policy (configurable). Session data is in-memory only — no persistent conversation storage. Assessment results exist only for the duration of the session.
> - **Isolation**: Multi-tenant design uses per-user delegated tokens. Session keys include `tenant_id + user_id`, so one user's session data never leaks to another. Each assessment runs in its own session context."

---

### Q: Identity & auth model (users, apps, service principals, tool boundaries)?

**ANSWER (Velen / YJ):**

> "Three authentication layers:
>
> 1. **User → Agent**: Entra ID OAuth2 authorization code flow. Users authenticate via browser redirect. The agent receives a user token with their identity and permissions baked in.
>
> 2. **Agent → Azure services**: Managed Identity. Container Apps → Azure OpenAI, Key Vault, App Insights — all via Managed Identity. Zero API keys in config.
>
> 3. **Agent → Graph API**: Delegated permissions using the user's OAuth token. The agent acts on behalf of the user — it can only query data the user is authorized to see. For app-level access, we use client credentials stored in Key Vault.
>
> **Tool boundaries**: Assessment tools (1–4) are read-only Graph queries. The remediation tool (5) calls Azure OpenAI but produces only text output — it cannot execute the scripts it generates. The Fabric tool (8) is write-only to a lakehouse — it cannot read other tenants' data. The agent's system prompt explicitly states: 'You are read-only. You NEVER make configuration changes.'
>
> We also support **consent revocation** — an admin endpoint (`POST /auth/revoke-consent`) can revoke a user's consent and disable the service principal in their tenant."

---

### Q: Threats addressed (prompt injection, exfiltration, unsafe automation)?

**ANSWER (Velen / YJ):**

> "We address the three big agent threats:
>
> 1. **Prompt injection**: Azure AI Content Safety screens all user inputs before they reach the LLM. The system prompt includes explicit instructions to ignore override attempts. Input validation middleware enforces length limits and character set restrictions. Even if injection succeeds, the agent can only call read-only tools — there's nothing destructive to exploit.
>
> 2. **Data exfiltration**: PII redaction strips identifiable data before it reaches Azure OpenAI — tenant IDs, emails, UPNs, IPs are all replaced with tokenized placeholders. The agent cannot make outbound network calls except to its configured Azure endpoints. Tool outputs are logged to audit but with redaction applied.
>
> 3. **Unsafe automation**: The agent is **assessment-only**. It generates PowerShell scripts in its remediation plans, but it cannot execute them. The disclaimer watermark on every output explicitly states: 'Review with your security team before implementing.' There is no tool that can write to or modify the customer's M365 tenant.
>
> Additionally: every action is logged in the immutable audit trail — what was asked, which tools ran, what was returned. Security admins can query this via KQL in App Insights."

---

### Q: Governance controls: policy, approvals, auditability, and admin controls?

**ANSWER (Velen / YJ):**

> "Four governance layers:
>
> 1. **Audit trail**: Every agent action logged with timestamp, session ID, user identity, tool called, redacted I/O, and reasoning. Stored in App Insights `customEvents`. RBAC-protected — only security admin roles can query audit logs.
>
> 2. **Admin controls**: Multi-tenant is gated by an allowlist (`ALLOWED_TENANTS` environment variable). If a tenant isn't on the list, the agent refuses to assess. Consent revocation endpoint lets admins de-authorize any tenant instantly.
>
> 3. **Content Safety policy**: Configurable block thresholds (`CONTENT_SAFETY_BLOCK_THRESHOLD`). If any category (violence, hate, self-harm, sexual) exceeds the threshold, the response is blocked and a safe fallback is returned.
>
> 4. **Operational policy**: Branch protection on `main` requires passing CI (688 tests + 80% coverage gate). OIDC federation means no long-lived secrets. Managed Identity means no API keys in config. Key Vault access is RBAC-controlled.
>
> The system prompt itself is a governance control — it's not hardcoded behavior but configurable instructions that define the agent's boundaries. Enterprise admins can modify the system prompt to add org-specific compliance requirements without changing code."

---

### Q: Responsible AI: grounding, transparency, human-in-loop, limitations?

**ANSWER (Velen / YJ):**

> "Our RAI posture:
>
> 1. **Grounding**: The agent reasons over **structured data from Microsoft Graph API** — not web searches or unverified sources. Secure Scores come from Microsoft's own security engine. Defender coverage percentages are real onboarded-device counts. The LLM is used only for the remediation plan synthesis — and even there, it's grounded in the structured assessment output.
>
> 2. **Transparency**: Every remediation recommendation includes a **confidence score** (high/medium/low) based on data completeness and remediation standardness. Every AI-generated output carries the disclaimer watermark: 'Generated by SecPostureIQ (AI-assisted) — review with your security team before implementing.'
>
> 3. **Human-in-the-loop**: The agent produces plans — humans execute them. It cannot make changes. The PowerShell scripts it generates must be reviewed and run by an admin. There is no automated execution path.
>
> 4. **Known limitations**: We disclose that PIM data may be unavailable with app-level tokens (requires admin consent for delegated access). We disclose that remediation impact scores are AI-estimated, not Microsoft-calculated. We disclose that mock data is used when Graph credentials aren't available. All of this is documented in our README under 'Known Limitations.'"

## Category 5: Copilot SDK & Agent Design Craftsmanship (implicit — storytelling: 15 pts)

### Q: Agent loop design (planning, tool choice, execution, reflection).

**ANSWER (Velen / YJ — agent design):**

> "The architecture leverages the Copilot SDK's agentic runtime for planning and tool orchestration:
>
> 1. **Planning**: The Copilot Agent Runtime reads the system prompt (which lists all 8 tool capabilities and behavioral instructions) and the user's query. It decides which tools to call and in what order. We don't hardcode the orchestration — the runtime does.
>
> 2. **Tool choice**: The runtime matches the user's intent to registered tools based on their descriptions and parameter schemas. For a 'full assessment' query, it chains: secure_score → defender → purview → entra → remediation_plan → adoption_scorecard. For a targeted query like 'check my Defender coverage,' it calls just that one tool.
>
> 3. **Execution**: Each tool adapter in `main.py` bridges the SDK's `ToolInvocation` to our tool implementation, calls the Graph API (or Azure OpenAI for remediation), and returns a `ToolResult`. All within OpenTelemetry trace spans.
>
> 4. **Reflection**: The remediation plan and adoption scorecard tools take **all prior tool outputs as context** (`assessment_context` parameter). This means the agent synthesizes across all findings — it doesn't just report each tool independently. The scorecard computes cross-workload adoption percentages and identifies the top 5 gaps by priority.
>
> We also built **two fully parallel interfaces** with the same tool parity:
> - The **Copilot SDK CLI agent** (`src/agent/main.py`) — uses `CopilotClient`, `Tool` registration, and the agent runtime for planning.
> - The **FastAPI Web App** with 3 chat modes: keyword-based (`/chat`), LLM-powered SSE streaming with function calling (`/chat/stream`), and **Chainlit conversational UI** (`/chat-ui`). Both the SSE streaming and Chainlit modes use Azure OpenAI function calling directly.
>
> Both interfaces share the same 8 tools, same system prompt, same middleware stack, and the same tool schemas (`src/tools/definitions.py`). The web app's SSE streaming mode uses Azure OpenAI with function calling for intelligent tool selection — the LLM decides which tools to call just like the Copilot SDK runtime does. This means both paths are LLM-powered, just through different orchestration mechanisms."
